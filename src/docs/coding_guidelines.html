Coding Guidelines

(1) Run all unit tests after making changes to see whether anything has broken. You can do this using the runtests Ant target.

(2) Consider writing junit tests.

(3) Consider using assertions.

(4) Avoid circular dependencies between packages. I.e. if any classes in package A import classes from package B, then classes in package B should not import classes from package A.

(5) Follow the package dependency rules. The checkdep ant script tests compliance. To run it type: ant -f checkdep.xml The script and the html page are generated by the class experimental.GenerateDependenciesXmlAndHtml.

(6) Check the javadoc Ant target runs without errors or warnings.

(7) Javadoc comments. Add a couple of sentences saying what the class does and the reason for its addition with a date. Also use the @author tag if you add a new class or make a significant change to an existing one.

(8) Some of the GUI classes have been created using the NetBeans GUI editor. The .java files for these classes have corresponding .form files. Within the java files are protected sections of code, if you edit these files, please do not alter these sections by hand.

(9) Use Code Formatters with care. Avoid reformatting whole files with different formatting schemes when you have only changed a few lines. There is an target to format the code. It uses Jalopy. Note, the GUI classes created with the NetBeans GUI editor are excluded from this format target. This is because Jalopy breaks the protected regions. These files can be formatted using the default formatting scheme in NetBeans.

Submitting Patches

(1) Run CVS diff with output in 'unified' format. (You can do this in Eclipse by selecting Team -> Create Patch.)

(2) Submit patches using the patch or bug tracker on the freerails sourceforge page.

(3) Give the patch a meaningful filename (e.g. bug910902_patch1.diff).

(4) Small patches that fix one thing, add one feature, or clean up one aspect of the code are generally the easiest to manage.
